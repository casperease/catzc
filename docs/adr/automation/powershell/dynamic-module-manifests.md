# ADR: Dynamic module manifests — the PowerShell layer over open-closed-architecture

## Rules: ADR-AUTO-MANIFEST

### Rule ADR-AUTO-MANIFEST:1

The PowerShell module system realizes [open-closed-architecture](../open-closed-architecture.md) (`ADR-AUTO-EXTEND`) mechanically: module
manifests are a generated projection of the filesystem, and the one seam convention cannot guard — a duplicate function name — fails the
import before anything loads.

- [Why this matters for PowerShell specifically](#why-this-matters-for-powershell-specifically)

### Rule ADR-AUTO-MANIFEST:2

Never hand-maintain manifests: `.psd1` files are generated by `New-DynamicManifest`. The filesystem is the single source of truth for what a
module contains and exports.

- [Why this matters for PowerShell specifically](#why-this-matters-for-powershell-specifically)

### Rule ADR-AUTO-MANIFEST:3

Generated manifests are gitignored, never committed. `New-DynamicManifest` recomputes each module's `<Module>.psd1` from the filesystem on
**every** import, writing only when the content differs (and deleting it when a module has no `.ps1` files left), so the manifest is a pure
derived artifact — `automation/**/*.psd1` is gitignored. The only committed `.psd1` files are the vendored third-party manifests under
`automation/.vendor/` (negated in `.gitignore`), which the platform does not generate.

- [Manifests are a regenerated, gitignored artifact](#manifests-are-a-regenerated-gitignored-artifact)

### Rule ADR-AUTO-MANIFEST:4

Every function name resolves to exactly one definition. Two automation modules cannot export the same name, and no automation function may
shadow an already-imported vendored or built-in command — `Import-AllModules` fails fast on either before any module loads, since PowerShell
would otherwise let the module imported last silently win. Beyond exports, every function name — public or private — must be defined in
exactly one file across the repository, verified by an integrity test.

- [One global, collision-free function namespace](#one-global-collision-free-function-namespace)

## Context

[open-closed-architecture](../open-closed-architecture.md) fixes the doctrine: grow the system by adding files and folders, never by editing
the infrastructure that discovers them. This ADR is the PowerShell layer under it — the manifest and namespace mechanics that make the
doctrine hold in this module system.

PowerShell module systems typically require hand-maintained manifests, explicit import lists, or registration steps. Each new function means
editing a `.psd1`, updating an export list, or adding a line to a loader script. These edit-to-extend workflows create merge conflicts,
stale manifests, and "I added the function but forgot to export it" bugs. The bootstrap module eliminates that entire class of problems by
generating manifests dynamically from the filesystem.

### Why this matters for PowerShell specifically

**No linker catches missing exports.** In compiled languages, a missing export causes a build error. In PowerShell, a function that exists
in a file but is missing from `FunctionsToExport` silently does not load. The caller gets a confusing "command not found" at runtime — often
in CI, often after the PR already merged. Dynamic manifest generation eliminates this failure mode entirely.

**Hand-maintained lists rot.** A `.psd1` manifest with an explicit `FunctionsToExport` list is a second source of truth that must stay in
sync with the actual files. Every rename, move, or delete requires updating both the file and the manifest. People forget. Reviews miss it.
The list drifts. Filesystem-driven discovery means the filesystem _is_ the manifest — there is no second source of truth to keep in sync.

**Merge conflicts on shared files.** When every new function requires editing a central manifest or loader, two developers adding functions
in parallel will conflict on that file. The conflict is trivial but annoying, and it blocks CI. Convention-based discovery means parallel
additions never touch the same file.

**Onboarding cost compounds.** "To add a function, create a file, then edit the manifest, then update the export list, then add it to the
loader" is four steps where one would do. Each extra step is a place a newcomer can get stuck. "Drop a `.ps1` file in the folder" is
self-evident.

### Manifests are a regenerated, gitignored artifact

A module's `.psd1` is not source — it is a projection of the module's `.ps1` files, recomputed every time the importer runs.
`Import-AllModules` calls `New-DynamicManifest` for each module directory immediately before `Import-Module`, deriving
`<Module>/<Module>.psd1` directly from the current filesystem: every root `*.ps1` (excluding `*.Tests.ps1`) becomes an exported function,
every `private/*.ps1` becomes a non-exported `NestedModules` entry. The file is written only when the recomputed content differs — a clean
tree costs a read per module and no write, which is also what lets Test-Automation's parallel workers import concurrently without racing the
manifest files. A directory that has lost all its `.ps1` files has its stale manifest deleted in the same loop. The manifest is never read
as an input and never hand-edited; the filesystem of `.ps1` files is the single source of truth, and the manifest is its output.

Because the manifest is regenerated from scratch on every import, committing it would be committing a build output. It would drift from the
`.ps1` files on any branch that adds, renames, or moves a function, and produce noisy, conflict-prone diffs that say nothing the directory
listing does not. So the generated manifests are gitignored, with one negation for the vendored case:

```gitignore
automation/**/*.psd1
!automation/.vendor/**/*.psd1
```

The exception is real source from the dependency's perspective: a vendored third-party module under `automation/.vendor/` ships its own
author-written `.psd1`, which the platform does **not** generate and which must be committed for offline reproducibility (see
[vendor-toolset-dependencies](vendor-toolset-dependencies.md)). Ours are derived; theirs are vendored source — which is exactly why one is
gitignored and the other is committed.

### One global, collision-free function namespace

Convention-as-source-of-truth carries one assumption: that a function name means exactly one thing. The export list is derived from file
names (`Get-Foo.ps1` exports `Get-Foo`) and every module imports into the global scope, so the whole repository shares a single function
namespace. Two modules that both export `Get-Foo` do not error — PowerShell lets the module imported last silently shadow the other, and a
caller gets whichever module sorted last. The same holds against the modules imported before ours: a function named like a vendored or
built-in command shadows it just as silently.

This is the one open/closed seam where "non-conforming code is invisible" does not hold — both definitions are discovered, and one quietly
wins. So the import enforces it directly. Before any module (or C# type) loads, `Import-AllModules` computes the names each module would
export and fails fast, naming every collision, when:

- two automation modules would export the same name, or
- an automation function would shadow an already-imported shipped command — one from a vendored module or a built-in. The shadow check is
  scoped to shipped commands by module path, so a user's unrelated session module never blocks the import.

The import-time guard scopes itself to the set the manifests export — module-root functions always, `private/` functions under
`-ExportPrivates` — because that is the set a load can actually break. The repository is stricter than that: an integrity test
(`Test-Automation.Tests.ps1`) parses every module file and fails if any function name, public or private, is defined in more than one file.
Reusing a private name across modules would not break an import (privates are module-scoped), but the repo forbids it anyway so the
one-function-per-file promise — `Get-Foo.ps1` is the file, anywhere in the tree — holds for private helpers too. `Test-FunctionDependency`
independently fails on a duplicate public name as well.

## Decision

Module manifests are generated by `New-DynamicManifest` from the filesystem on every import and never committed; the import fails fast on
any duplicate export or shadow of a shipped command; and an integrity test holds every function name — public or private — to exactly one
defining file.

### How this is enforced

- **`New-DynamicManifest`** — the single manifest generator, driven by `Import-AllModules`; there is no hand-edited manifest to drift.
- **`Import-AllModules`** — the pre-load collision and shadow check, failing the import with every collision named.
- **`Test-Automation.Tests.ps1`** + **`Test-FunctionDependency`** — the repository-wide unique-name integrity checks.
- **`.gitignore`** — `automation/**/*.psd1` with the `.vendor` negation keeps derived manifests out of git and vendored ones in.

## Consequences

- Adding a function never touches a manifest, an export list, or a loader — the file is the registration.
- The whole repository shares one function namespace, and it is collision-free by construction: a duplicate export — or a function that
  would shadow a vendored or built-in command — fails the import with a named error instead of silently shadowing.
- `git status` never shows generated manifests, and a branch switch never conflicts on them.
- The cost is that the manifest is invisible as an artifact: a reader who wants a module's export list reads the directory (which is the
  point), or lets an import materialize the `.psd1`.
